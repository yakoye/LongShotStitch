# Superpowers 开发约束 / Development Requirements

适用项目：LongShotStitch 长图拼接、裁剪与标注工具。

本文件是项目长期开发约束。具体版本设计放在 `docs/superpowers/specs/`，实施计划放在 `docs/superpowers/plans/`，版本结果放在 `docs/vX.Y.Z修改记录.md`。标注与属性界面的逐项执行规则见 `docs/ANNOTATION_UI_CHECKLIST.md`。

## 1. 约束优先级

1. 用户当前明确要求；
2. 本文件的项目长期约束；
3. 已确认设计文档和实施计划；
4. 通用 Web 开发习惯。

发生冲突时采用更具体、更新且经用户确认的规则，并在设计文档中记录原因。

## 2. 版本与交付

- 使用 `v1.x.x`：中间位表示功能迭代，末位表示缺陷修复。
- 每次开发必须明确本版范围，不能混入未确认需求和无关重构。
- 页面标题、`APP_VERSION`、可见版本、README、修改记录、测试和 ZIP 版本必须一致。
- `LongShotStitch.html` 与 `index.html` 必须完全一致。
- ZIP 命名为 `LongShotStitch-vX.Y.Z.zip`。
- 每次交付必须给出准确 Git 命令，提交信息不得使用模糊的 `update`。

## 3. 开发流程

- 修改前先阅读当前实现、现有文档、相关测试和最近版本记录。
- 新功能或交互变化先写设计文档，再写实施计划。
- 缺陷修复先确认复现路径和根因，禁止只对表面症状打补丁。
- 能自动验证的需求先添加失败测试，再修改实现。
- 每次只验证一个明确假设；连续多次无效修复时应停止并重新检查架构。
- 已经稳定的功能尽量局部修改，不得为了代码“更漂亮”而大范围重写。

## 4. 功能边界与对象模型

- 修改标注时不得破坏拼接、整体裁剪、单图裁剪、分割线和导出。
- 修改裁剪、拼接或导出时不得破坏标注对象的再次编辑。
- 所有标注、粘贴图片和浮动选区都必须保持对象化，可再次选择、移动和修改。
- 新功能必须兼容撤销、重做、工程保存、工程重开和导出。
- 不得通过自动压平画布规避对象编辑问题；只有用户主动执行扁平化时才允许压平。
- 水印保持全局单对象语义。

## 5. 标注交互一致性

- 工具模式点击已有对象时，优先选择或编辑已有对象；点击空白才创建新对象。
- 单击对象用于选择和变换；属性和文字内容编辑不得抢占基本拖动操作。
- 连续工具（铅笔、荧光笔、自由马赛克、橡皮）绘制过程中不显示选择框。
- 选择框、端点、缩放点和旋转点必须与对象真实变换一致。
- 选中对象后，其属性必须稳定对应当前对象，不能因数组索引或重绘丢失。
- 单击空白、再次点击选择工具或按 Esc 均可取消选择。
- Shift 约束、图层、复制、删除等通用操作应在适用对象间保持一致。

## 6. 桌面端与手机端

- 桌面端和手机端必须分别设计、分别测试，不能把桌面布局简单压缩到手机。
- 桌面端优先：顶部工具栏 + 左侧模式/属性栏 + 中间画布。
- 手机端优先：顶部简化操作 + 底部模式/工具栏 + 可收起紧凑属性浮层。
- 手机端工具顺序和功能语义必须与桌面端一致。
- 手机端单击对象不得自动弹出遮挡对象的大属性面板。
- 触摸控制点要有足够命中区域；浮层不能越界或被浏览器底栏遮挡。
- hover 必须有触摸替代；拖动、双指缩放、点击外部关闭和 Esc 行为分别检查。

## 7. UI 与组件规范

- 界面保持简洁、克制、紧凑；工具页不得堆放大段说明卡。
- 常见操作优先图标 + 短 tooltip，不重复解释默认含义。
- 页面不得出现 TODO、mock、迁移状态、实现细节或开发者提示。
- 按钮、输入框、select、popover、dialog、tooltip 和菜单应使用统一的高度、圆角、边框、字体和状态。
- 数字输入框按内容宽度设计，不允许无意义地撑满整行。
- 属性优先单行排列；空间不足时在组件内部换行或滚动，不得撑破页面。
- 危险操作使用红色；普通操作使用中性色；主强调色保持单一。
- 控件必须具备默认、hover、active、disabled、selected 等必要状态。
- 公开文案面向普通用户，开发说明写入 README 或 docs。

## 8. 文字工具

- Canvas、选择框和行内编辑器必须共享同一字体、字号、行高、内边距和测量逻辑。
- 编辑时 Canvas 不重复绘制同一文字，避免重影。
- 文字编辑框必须贴合对象，不得无故占满手机屏幕。
- 单击已有文字只选择；双击或显式编辑按钮才修改内容。
- 文字背景默认透明，不提供多余背景框。
- 旋转后的文字、选择框和旋转手柄必须共享同一视觉中心。

## 9. 状态与数据

- 设置、工具子类型、折叠状态和用户明确需要保留的偏好应持久化。
- UI 出现不代表功能完成；保存、读取、删除、下载和恢复必须真实执行。
- 文件格式错误、权限失败、超大图片和空数据应有轻量明确反馈。
- 离开页面前保护未保存工程，避免刷新或返回造成无提示丢失。

## 10. 测试与验证

每次开发至少执行：

- JavaScript 语法检查；
- `tests/smoke_check.js`；
- `tests/static_check.js`；
- 与本次修改相关的专项测试；
- `LongShotStitch.html` 与 `index.html` 一致性检查；
- ZIP 解压后二次测试。

涉及 UI 和交互时还必须检查：

- 桌面端和手机端；
- 创建、按下、移动、松开、取消；
- 选择、属性、拖动、缩放、旋转；
- 撤销、重做、保存、重开、导出；
- 点击外部、Esc、边界、遮挡和快速重复操作。

不得仅凭阅读代码声称修复。无法自动化的项目必须在交付说明中明确列为人工验证项。

## 11. 发布检查清单

- [ ] 本版需求逐项对应实现或明确延期；
- [ ] 没有引入无关重构；
- [ ] 桌面端主流程可用；
- [ ] 手机端主流程可用且无横向溢出；
- [ ] 所有标注工具可创建和再次编辑；
- [ ] 属性面板不遮挡基本变换操作；
- [ ] 输入框、按钮和菜单样式统一；
- [ ] 撤销、重做、工程保存与导出正常；
- [ ] 所有自动测试为最新一次完整运行结果；
- [ ] 入口、版本、文档和压缩包一致；
- [ ] Git 提交命令准确描述本次修改。

## 12. 核心原则

> 不做“看起来完成”的界面；做一个真实用户长期使用时稳定、紧凑、自然、不会误触、不会丢状态、不会被浮层挡住操作的工具。
